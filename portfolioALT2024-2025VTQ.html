UPD V1 - Script Python de génération des rapports (sans API ni interface) C1 C2 C6
Période : 2024-2025
Contexte

Livrer un script Python autonome pour produire les rapports réglementaires UPD à partir de sources Excel/CSV hétérogènes, avec un temps d'exécution réduit et des contrôles de qualité intégrés. Aucune API Flask ni interface Symfony demandée sur cette V1.

Objectifs
  • Concevoir un pipeline fiable (lecture, nettoyage, règles métiers, export).
  • Optimiser les performances (lecture/écriture, opérations vectorisées).
Actions clés
  • Implémentation Pandas : normalisation de colonnes, jointures, détection de doublons/incohérences.
  • Optimisations : indexation, vectorisation, réduction des allocations mémoire.
  • Journalisation et messages d'erreur exploitables pour les utilisateurs métiers.
Résultats & Impact
  • Génération reproductible et baisse d'erreurs manuelles.
  • Temps d'exécution amélioré grâce aux optimisations Pandas/IO.
Savoirs (connaissances)
  • Modèles de données tabulaires, formats Excel/CSV, validation.
  • Principes d'algorithmique pour le traitement de données (complexité).
Savoir-faire (techniques)
  • Écriture de pipelines Pandas performants (merge, groupby, apply -> vectorisation).
  • Profiling/optimisation (mémoire, IO), export Excel fiable.

Portail UPD V2 - Web (PHP 8 / Symfony 6.4) C1 C2 C6
Période : 2025
Contexte

Refonte du portail pour piloter la création des datasets et déclencher les traitements pour générer un fichier final qui contient les données de ventes en Europe ou Hors Europe pour les produits qui ont une authorisation de ventes sur le téritoire Européen. Avec une gestion des phases, upload multipart, appels à l'API Python, suivi d'avancement et administration de l'application.

Objectifs
  • Interface claire pour les métiers, fiabilisation des envois.
  • Asynchronisme côté front et backend (files de messages) pour ne pas bloquer l'IHM.
  • Autonomie des utilisateurs.
  • Reproductibilité pour faciliter l'audit ou le vérification de données.
Actions clés
  • Intégration HttpClient (POST/DELETE), puis offloading via Symfony Messenger (handlers, queues).
  • Notifications d'état, relances, logs et gestion d'erreurs.
  • Optimisations des requêtes SQL et pagination.
Résultats & Impact
  • Le projet est toujours en cours de développement.
  • Autonomie des métiers.
  • Moins d'incidents (reprises automatiques, journalisation).
Savoirs (connaissances)
  • Gestion d'états asynchrones, fonction avancer en Symfony, fonction plus avancer en Flask.
  • Qualité logicielle : validation pharmaceutique, sécurité applicative.
Savoir-faire (techniques)
  • Contrôleurs/formulaires Symfony, HttpClient, Messenger/Handler.
  • Conception d'écrans UX pour le suivi de dataset.
  • Utilisation d'un WebSocket et de WebHook depuis l'API Flask

  • Workflow dataset : clarifie qui fait quoi et quand (phases 1->4), distingue automatisé vs manuel, matérialise les gates qualité et les décisions Oui/Non.

Définition - « Dataset » (contexte UPD)

Dans ce projet, un dataset désigne l'ensemble structuré de fichiers sources (Excel/CSV, etc.) et métadonnées nécessaires à la génération d'un rapport UPD : fichiers mensuels, référentiels, règles de validation/qualité et paramètres d'extraction. L'intérets est de pouvoir lancer le traitement Europe ou Hors Europe avec une des identiques et reproductible.

  • Il est préparé côté portail (UPD V2) avant envoi au traitement.
  • Il doit satisfaire des contraintes de schéma, d'unicité et de complétude.
  • Le workflow de création de Dataset est dans les Traces de ce projet.
